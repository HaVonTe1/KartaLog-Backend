# AGENTS.md – Persistence Adapter Guidelines
---
## OVERVIEW
- **Purpose**: Bridge domain model to relational storage using JPA.
- **Tech**: Kotlin + Spring Data JPA, Maven module `app/adapter/persistence`.
- **Scope**: Entity definitions, Spring repositories, mappers, transactional adapters, logging.

## JPA ENTITIES
- Annotated with `@Entity`, `@Table(name = "...")`.
- Primary key: `@Id @GeneratedValue` (Long or UUID).
- Immutable fields `val`; mutable only when required.
- Implement `fun toDomain(): DomainObject` converting JPA row to domain model.
- Use Kotlin data classes where possible; avoid `var` for persistence‑only fields.
- Add `@Column(nullable = false)` for non‑null columns; use nullable Kotlin types for optional columns.

## REPOSITORY ADAPTERS
- Spring Data interfaces extend `JpaRepository<Entity, Id>`.
- Marked `@Repository` and injected via constructor (`@Component`).
- All public methods annotated `@Transactional` (read‑only = `readOnly = true`).
- Adapter class implements domain `XXXRepository` and delegates to Spring repo.
- Keep adapters thin: translate to/from domain, handle exceptions, log.

## MAPPERS
- Dedicated mapper objects/functions (`EntityMapper`) for bidirectional conversion.
- Use extension functions: `entity.toDomain()` and `domain.toEntity()`.
- Map only necessary fields; avoid lazy‑loading side effects.
- For collections, map each element individually.

## LOGGING
- One logger per class: `private val log = LoggerFactory.getLogger(this::class.java)`.
- Log entry/exit of public adapter methods at `DEBUG`.
- Log repository errors with stack trace at `ERROR`.
- Include entity identifiers in log messages for traceability.

## NOTES
- Keep JPA annotations **module‑local**; domain models stay framework‑agnostic.
- Favor constructor injection; avoid field injection.
- Do not place business logic in entities or repositories.
- Run `mvn -q test` after changes; ensure `mvn detekt:check` passes.
- Update this file when adding new entities, repo interfaces, or mapper conventions.
